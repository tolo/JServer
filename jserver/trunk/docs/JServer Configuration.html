JServer 2.2 - Configuration

 

1.      Introduction

1.1. Dynamic, static or hybrid

In the past, most JServers have been built primarily as static servers, i.e. all functionality and structure has been set at compile time. Since version 1.3.1, however, the support for adding functionality dynamically has been extended and made more accessible. This is even more so the case with the release of version 2.0.

JServers can now in essence be run in three distinct modes; strictly dynamic (all components are added at runtime), static (all components added at compile time) or hybrid (some components are added at compile time, some at runtime). When running the server strictly dynamically, the JServer class itself is used as main class. For this purpose, JServer is equipped with a main method to which a set of parameter may be specified (see next chapter). When running in static or hybrid modes, a custom main class is used for static creation and registration of component. Since version 2.0 Build 20041213, however, custom main class implementations may call the method JServer.createServer to make use of the JServer parameter functionality.

When adding components dynamically, the JServer will make sure that the component is registered as a child component in the parent component it is defined under. The JServer will also start the dynamic components in a top-down pattern.

 

1.2. JServer start-up parameter

The following parameters may be specified for the main method, as well as the createServer method, of JServer:

  • -name <name> - The name of the server.
  • -empty - Creates an empty server, i.e. no default components are added.
  • -noDefaultAppenders - Creates a default configuration server, adds no appenders to the LogManager.
  • -conf <config file> - Uses the specified configuration file.
  • -confClass <config class> - Uses the specified class as PersistentPropertyStorage implementation in the PropertyManager.

 

2.      Configuration file syntax

2.1. Introduction

As of version 2.0, the default (and the preferred) configuration data source is a XML file, rather than the simple text file format used in previous versions. Besides the obvious differences in file format, one of the major differences between the 2.0 XML file and the 1.X text file, is that only properties that are not using their default value are now saved to the 2.0 XML file.

This chapter will describe the syntax of the XML configuration file. Chapter 4 contains examples of a few different configuration files.

2.2. Basic structure


The syntax of the configuration file consists of three primary tags, JServer, Component, and Property. These tags will be described individually in the following chapters.

2.3. JServer


The JServer tag is the root tag of the XML document. Only one attribute may be specified for this tag - name - i.e the name of the server. In all other respects, the JServer tag is treated similarly to the Component tag.

2.4. Component


Component tags represent SubSystem or SubComponent components in a JServer. The following attributes may be specified for a component tag:

  • name - the name of the component (the full name of the component is determined by the names of the parent component tags).
  • class - the class implementing the component. This attribute is only necessary if the component is to be loaded dynamically, i.e. if the class is not statically compiled into the server.

Component tags may contain sub tags of the type property and component.


Example:

<JServer name="Client">

      <Component name="SendController">

            <Component name="MessagingManager">

                  <property name="log level" value="Debug"/>

            </Component>

      </Component>

</JServer>

 

2.5. Core component tags


A set of core components (common JServer components) have reserved tag names which must be used when configuring these components. If the server is run strictly dynamically, these names are used to create the components. Although they have different names, all of them are in essence component tags and follow the same rules.
The tags are as follows:

  • EventQueue - The event queue (com.teletalk.jserver.event.EventQueue) accessible through methods like JServer.getEventQueue(). See configuration of EventQueue in chapter 3.4.1 for more information.
  • LogManager -  The log manager (com.teletalk.jserver.log.LogManager) accessible through methods like JServer.getLogManager(). See configuration of LogManager in chapter 3.6.5 for more information.
  • MainHttpServer - The main http server (com.teletalk.jserver.tcp.http.MainHttpServer) accessible through the method JServer.getMainHttpServer(). See configuration of HttpServer in chapter 3.19.1 for more information.
  • PeriodicActionManager - The periodic action manager (com.teletalk.jserver.periodic.PeriodicActionManager) accessible through methods like JServer.getPeriodicActionManager(). See configuration of PeriodicActionManager in chapter 3.10.1 for more information.
  • RmiManager - The RMI manager (com.teletalk.jserver.rmi.RmiManager) accessible through methods like JServer.getRmiManager(). See configuration of RmiManager in chapter 3.15.1 for more information.
  • ConfigurationManager - The configuration (property) manager (com.teletalk.jserver.property.PropertyManager), accessible through methods like JServer.getPropertyManager(). See configuration of PropertyManager in chapter 3.12.1 for more information.
  • RmiAdministration - Not really an actual component, but rather the setter of a flag to indicate that RMI administration should be enabled. This is only necessary when the server is run strictly dynamically.
  • LoadManager - Creates the load manager (com.teletalk.jserver.load.LoadManager), accessible through methods like JServer.getLoadManager(). See configuration of LoadManager in chapter 3.5.1 for more information.
  • StatiticsManager - Creates the statistics manager (com.teletalk.jserver.statistics.StatisticsManager), accessible through methods like JServer.getStatisticsManager(). See configuration of StatisticsManager in chapter 3.17.1 for more information.
  • SpringApplicationContext - Creates the Spring application context (com.teletalk.jserver.spring.SpringApplicationContext), accessible through the method JServer.getSpringApplicationContext(). See configuration of SpringApplicationContext in chapter 3.16 for more information.
  • SnsClientManager - Creates the SNS client manager (com.teletalk.jserver.net.sns.client.SnsClientManager), accessible through methods like JServer.getSnsClientManager(). See configuration of SpringApplicationContext in chapter 3.8.1 for more information.
  • SnsManager - Creates the SNS server side component (the actual SNS) (com.teletalk.jserver.net.sns.server.SnsManager), accessible through methods like JServer.getSnsManager(). See configuration of SpringApplicationContext in chapter 3.9.1 for more information.

A custom implementation class may be specified (attribute class) for most of the core components, except for EventQueue, LogManager, ConfigurationManager and RmiAdministration.

 

2.6. Property

Property tags assigns values to properties (settings) of components in a JServer. The following attributes may be specified for a property tag:

  • name - the name of the property (the full name of the component is determined by the names of the parent component tags).
  • value - the value of the property.

The value of a property may also be specified through the body of the property tag or set by one or more value child tags. The latter is useful for properties that may have more than one value.

Example:

<SubSystem name="KahunaSystem">

<property name="property1" value="value"/>

<property name="property2">value</ property >

<property name="adresses">

           <value>1.2.3.4:1111</value>

           <value>1.2.3.4:2222</value>

</property>

</SubSystem>

2.7. Property meta data

For all property tags, the option to specify extra meta data exists. Meta data is specified through one or more metadata tags. The following attributes may be specified for a metadata tag:

  • name - the name/key of the meta data entry.
  • value - the value meta data entry.

The value of a meta data entry may also be specified through the body of the metadata tag.

2.8. Types of properties


Properties come in a variety of different flavors for different types of values, for instance string (StringProperty), numerical (NumberProperty) or boolean (BooleanProperty) values. For certain properties, multiple values may be specified. These are by default MultiStringProperty, MapProperty (values on the format <key>=<value>) and IpAndPortProperty/
TcpEndPointIdentifierProperty (values on the format <ip/host>:<port>).
Another type of property is the EnumProperty, which uses a fixed set of possible values. These values may be referred to as a name or by an index. An example of this is for instance the property log level of SubComponent, when for instance the log level info may be set by specifying the string Info or the number 2.

3.      Component configuration

3.1. Terminology etc


Properties that are non-persistent, i.e. read-only and strictly used for presentation are marked as italic and colored gray.

Properties that are only possible to set in the configuration file prior to server start are colored blue.

Properties that trigger a restart of the parent component are colored as red.


Note on default values: Other property default values than those stated here may be used on certain server implementations due to hard coding etc.

 

3.2. Common configuration

All components of a JServer are based on the classes SubComponent or SubSystem (which is based on SubComponent). This means that all components also inherit the properties of those two base components. These properties are described below:

3.2.1.     SubComponent

Properties:
  • status - The status of the component. May be one of the constants ENABLED, INITIALIZING, CREATED, REINITIALIZING, SHUTTING_DOWN, DOWN, ERROR, CRITICAL_ERROR, DESTROYED.
  • logLevel - The log level of the component. Valid values are “All” (0), “Debug” (1), “Info” (2), “Warning” (3), “Error” (4), “Fatal” (5), “Off” (6). Default value is Info.
  • errorReason - The last reason of an error in the component.

 

3.2.2.     SubSystem

SubSystem inherits the properties of SubComponent.

Properties:
  • startTime - The time when this subsystem was started.
  • maxRestarts - The maximum number of attempts that should be made to restart the subsystem when an error occurs. The minimum value of this property is 0. Default value is 5.
  • restartInterval - The interval in milliseconds between restarts. The minimum value of this property is 100 ms. Default value is 5000 ms.
  • restartCounter - The number of times the subsystem has been restarted.
  • keySystem - Boolean property indicating if this subsystem is a key system. If this flag is set to true, a fatal error will be logged if an error occurs in this subsystem and it cannot be restarted (status CRITICAL_ERROR). Default value is false.

 

3.3. Top system configuration - com.teletalk.jserver.JServer

JServer inherits the properties of SubSystem.

Properties:
  • activeThreadCount - The number of active threads in this server (belonging to the top thread group or one of it’s children).
  • jServerVersion - The JServer version.
  • serverVersion - The user defined version of the server.
  • checkInterval - The interval in milliseconds between two consecutive SubSystem checks. The minimum value of this property is 1000 ms (although it is advisable to use a considerably higher value). Default value is 30000 ms.
  • currentDirectory - The current directory of the server process.
  • javaVMInfo - Java VM information.

 

3.4. Components in package com.teletalk.jserver.event

3.4.1.     EventQueue

EventQueue manages the main event queue of a JServer. EventQueue inherits the properties of SubSystem. EventQueue also contains a ThreadPool (named EventDispatcherPool).

Properties:
  • maxQueueSize - The maximum size of the queue that holds event objects that are waiting to be processed. Default value is 10000. A value of 0 or less means that no maximum limit will be imposed.

 

3.5. Components in package com.teletalk.jserver.load

3.5.1.     LoadManager

LoadManager handles load monitoring of a JServer. LoadManager inherits the properties of SubSystem.

Properties:
  • loadThreads - The threads currently registered for reporting load.
  • load - The total load of all registered threads.
  • loadUpdateInterval - The interval in milliseconds at which the property indicating load will be updated. Min value is 100 ms. Default value is 500 ms.
  • threadCheckInterval - The interval in milliseconds at which the LoadManager will check for dead threads and reevaluate load. Min value is 1000 ms. Default value is 30000 ms.

 

3.6. Components in package com.teletalk.jserver.log

3.6.1.     AppenderComponent

AppenderComponent is an abstract base class of appender (log writer) implementations and inherits the properties of SubComponent.

Properties:
  • logLevelThreshold - The log level threshold of this appender component (not to be confused with the log level of this component!). This property determines the minimum log level of logging events that may be logged through this component. Valid values are “All” (0), “Debug” (1), “Info” (2), “Warning” (3), “Error” (4), “Fatal” (5), “Off” (6). Default value is All.

 

3.6.2.     PeriodicityAppenderComponent

PeriodicityAppenderComponent is an abstract base class for periodic appender (log writer) implementations and inherits the properties of AppenderComponent.

Properties:
  • periodicity - Enumeration property for the periodicity and cyclicity at which the current log will be exchanged. Valid values are “Daily (no cyclicity)” (0), “Daily (weekly cyclicity)” (1), “Daily (monthly cyclicity)” (2), “Weekly (no cyclicity)” (3), “Weekly (monthly cyclicity)” (4), “Weekly (yearly cyclicity)” (5), “Monthly (no cyclicity)” (6), “Monthly (yearly cyclicity)” (7). Default value is Daily (no cyclicity).
  • lastLogChange - The time of the last logchange.
  • nextLogChange - The time of the next logchange.
  • currentLog - The name of the current log.
  • logPrefix - The prefix of logs generated by this logger. Default value is the name of the server followed by an underscore.
  • logSuffix - The suffix of logs generated by this logger. Default value is empty string.

 

3.6.3.     FileAppenderComponent

FileAppenderComponent is a file based appender (log writer) component that inherits the properties of PeriodicityAppenderComponent.

Properties:
  • flushInterval - The interval in milliseconds between log file flushes. Default value is 5000 ms.
  • logFilePath - The base path of generated log files. Default value is current directory (empty string).
  • defaultLayoutPattern - The pattern used for the default layout - PatternLayoyt(ExtendedPatternLayout). Default value is %d  %-5p  [%t] - %m %S%n, which in essence means date and time, level, thread name, message and stack trace (if any). See Log4J documentation for more details on configuration of this property (see chapter 5). Note that the %S conversion character is JServer specific (through the use of the class com.teletalk.jserver.log.ExtendedPatternLayout), and is used for inclusion of stack traces.

 

3.6.4.     CriticalErrorAppenderComponent

CriticalErrorAppenderComponent is special appender (log writer) that dispatches critical erros to a Critical Error Log Manager (CELMa) or writes them to a file. CriticalErrorAppenderComponent inherits the properties of FileAppenderComponent, minus the properties logLevelThreshold, periodicity, lastLogChange, nextLogChange, currentLog, flushInterval and defaultLayoutPattern.

Properties:
  • remoteLoggingEnabled - Boolean property indicating if remote logging over HTTP is enabled. Default value is false.
  • remoteLoggingAddresses - The addresses (ip:port) to be used for logging over HTTP. Note that only one of the addresses will receive a log message. Default value is no addresses.
  • writeLocalErrorFiles - Boolean property indicating if local error files should be generated as a backup to remote logging (if remote logging is successful, no file will be generated event if this property is true). Default value is true.

 

3.6.5.     LogManager

LogManager is the core class for log management in a JServer and inherits the properties of SubSystem.

Properties:
  • maxQueueSize - The maximum size of the queue that holds log messages that are waiting to be processed. A value of 0 or less means that no maximum limit will be imposed. Default value is 10000.
  • getLocationInformation - Boolean property indicating if location information should be generated for logging events. Note that generation of location information may have an adverse effect on performance. Default value is false.
  • addToRootLogger - Boolean property indicating if this LogManager should be added as an appender in the root loggger, thus enabling logging through non JServer components. Default value is false.
  • rootLoggerLogLevel - The log level of the root logger. Default value is INFO.

 

3.6.6.     ManagedLoggerComponent

Subcomponent class for controlling the log level of one or more loggers that aren't defined through a subcomponent. The log level of this component is used to set the log level of the target loggers.

Properties:
  • managedLoggerNames - The names of the loggers that this component is to manage the log level for. The names specified here may be the complete name of a logger or a part of the logger chain - for example com.test.logger or com.test.

 

3.7. Components in package com.teletalk.jserver.net.acl

3.7.1.     DefaultAccessControlListHandler

DefaultAccessControlListHandler is the default access control list implementation. DefaultAccessControlListHandler inherits the properties of SubComponent.

Properties:
  • accessControlCheckEnabled - Flag indicating if access checks should be enabled or not. Default value is true.
  • configurationFile - The path of the access control list configuration file. Default value is "acl.xml".

 

3.8. Components in package com.teletalk.jserver.net.sns.client

3.8.1.     SnsClientManager

SnsClientManager is a component used to connect to an SNS server (SnsManager). This class inherits the properties of MessagingManager.

 

3.9. Components in package com.teletalk.jserver.net.sns.server

3.9.1.     SnsManager

SnsManager is a component implementing SNS (Service Name Service) functionality inherits the properties of MessagingManager.

Properties:
  • serverEntryLingerTime - The maximum time in minutes that a server entry will be kept in this SNS after the connection has been lost. Default value is 60.
  • manualServices - Storage for manually configured services. Format: <serviceName>=<address:port>,<address:port>. Default value is empty.

 

3.10.     Components in package com.teletalk.jserver.periodic

3.10.1.   PeriodicActionManager

PeriodicActionManager support handling and execution of periodic actions and and inherits the properties of SubSystem. PeriodicActionManager contains a ThreadPool (named PeriodicActionThreadPool).

Properties:
  • abnormalExecutionTime - The maximum time (in minutes) an action is allowed to execute before a critical error is logged.. Default value is 60.

 

3.10.2.   PeriodicAction

PeriodicAction is a base class for components implementing periodic actions. This class inherits the properties of SubComponent.

Properties:
  • periodicityType - Enumeration property for the periodicity type. Valid values are “INTERVAL” (0), “HOURLY” (1), “DAILY” (2), “WEEKLY” (3), “MONTHLY” (4). Default value is HOURLY.
  • periodicityDayOffset - The periodicity day offset. Default value is 0.
  • periodicityHourOffset - The periodicity hour offset. Default value is 0.
  • periodicityMinuteOffset - The periodicity minute offset. Default value is 0.
  • interval - The interval in milliseconds when periodicityType is INTERVAL. Minimum value is 100 ms. Default value is 60000 ms.
  • actionStatus - A string containing an action status description.
  • lastExecutionResult - A string representing the last execution result. lastExecutionStartTime - The time of the last execution.
  • timeOfNextExecution - The time of the next execution.
  • forcedExecution - Boolean property used to force the execution of this action.

 

3.11.     Components in package com.teletalk.jserver.pool

3.11.1.   ObjectPool

ObjectPool is a base component that suports pooling of objects. This class inherits the properties of SubComponent.

Properties:
  • minSize - The minimum size of the pool. The value of this property will only be used if max size if set. Default value is 0.
  • maxSize - The maximum size of the pool. If the value of this property is == 0, the pool is growable. Default value is 10.
  • objectExpirationTime - Expiration time in milliseconds of objects in the pool. Object expiration will be checked if this value is > 0, i.e. a value of 0 disables object expiration checking. Default value is 0 (unless specified in code).
  • cleanupInterval - The interval in milliseconds at which this pool is cleaned by the PoolCleaner. Default value is 60000.
  • statisticsEnabled - Boolean value indicating if statistical information should be calculated. This flag dictates if certain statistical properties will be visible. Default value is false.
  • averageCheckoutTime - The average time in milliseconds between check out and check in (statistics enabled must be true).
  • totalCheckouts - Total number of checkouts (statistics enabled must be true).
  • checkedOutItems - The number of checked out objects (statistics enabled must be true).
  • checkedInItems - The number of checked in objects (statistics enabled must be true).

 

3.11.2.   DefaultObjectPool

DefaultObjectPool is a default implementation of a simple object pool. This class inherits the properties of ObjectPool.

 

3.11.3.   ThreadPool

ThreadPool is a pool for threads and inherits the properties of ObjectPool.

 

3.11.4.   DBConnectionPool

DBConnectionPool is a pool for database connections and inherits the properties of ObjectPool.

Properties:
  • driver - The name (fully qualified class name) of the database driver used by this DBConnectionPool to create database connections.
  • databaseURL - The URL of the database which this DBConnectionPool should create connections to.
  • username - The username needed to access the specified database.
  • password - The password needed to access the specified database.

 

3.12.     Components in package com.teletalk.jserver.property

3.12.1.   PropertyManager

PropertyManager is the core configuration management component in the JServer. This class inherits the properties of SubSystem.

Properties:
  • calculationInteval - The interval in milliseconds at which properties owned by CalculatedPropertyOwners are recalculated (deprecated feature). Default value is 2000 ms.
  • saveInterval - The interval in milliseconds at which properties are saved to file (if needed). Default value is 15000 ms.
  • locale - String representation of the locale to use as default for the whole JVM (for instance sv_SE, en_US, etc). Note that the default locale is essential when representing certain properties as strings. Default value is the default locale of the JVM (as returned by the method Locale.getDefault().toString).

 

3.13.     Components in package com.teletalk.jserver.queue

3.13.1.   QueueManager

QueueManager is a component for distributed queue management and inherits the properties of SubSystem. QueueManager may contain up to two Queues (usually names in-queue and out-queue) and a DefaultQueueSystemCollaborationManager as child components.

Properties:
  • checkInterval - Interval in milliseconds at which checks a made. Default value is 10000 ms.
  • outitemWarningAge - The maximum age in milliseconds an item in the out queue can have before a warning is sent to the queue controller. Default value is 600000 ms.
  • maxInQueueSize - The maximum size of the in queue, -1 if the in queue has no maximum size. Default value is -1.
  • unsentResponseMaxAge - The maximum time (in hours) to keep an unsent response. Default value is 12 hours.

 

3.13.2.   DefaultQueueSystemCollaborationManager

DefaultQueueSystemCollaborationManager is a communication manager used by QueueManager and inherits the properties of TcpCommunicationManager. DefaultQueueSystemCollaborationManager contains a TcpServer (named TcpServer) as a child component.

Properties:
  • checkInterval - The interval in milliseconds at which checks are performed on all connections to other queue systems. Default value is 2500 ms.
  • statusMessageInterval - The interval in milliseconds at which status messages are dispatched to all remote queue systems connected to this queue system. Default value is 5000 ms.

 

3.13.3.   Queue

Queue (and also the class PriorityQueue) is a component representing a queue and inherits the properties of SubComponent. Queue may contain a queue storage as a child component, such as DefaultQueueStorage or FileDBQueueStorage.

Properties:
  • items - A list containing information about the items in the queue.

 

3.13.4.   DefaultQueueStorage

DefaultQueueStorage is the default queue persistence implementation and inherits the properties of ObjectPool.

Properties:
  • queueItemSavePath - The path when queue item objects are to be saved. Default value is the a directory under the current directory, named as the parent Queue and suffixed by “.storage”.

 

3.13.5.   FileDBQueueStorage

FileDBQueueStorage is a file database queue persistence component and inherits the properties of SubComponent.

Properties:
  • fileDatabaseDirectory - The directory where the file database will be stored. Default value is the a directory under the current directory, named as the parent Queue and suffixed by “.storage”.
  • dataFileBlockSize - The block size (in bytes) of the data file of the file database. Default value is 2048.
  • dataFileInitialBlockCount - The initial number of blocks in the data file. Default value is 1000.
  • indexFileInitialBlockCount - The initial number of blocks in the index file. Default value is 1000.

 

3.13.6.   InQueueControllerSystem

InQueueControllerSystem is a queue system controller component for queue systemts that contains an in queue. This class inherits the properties of SubSystem.

 

3.13.7.   OutQueueControllerSystem

OutQueueControllerSystem is a queue system controller component for queue systemts that contains an out queue. This class inherits the properties of SubSystem.

 

3.13.8.   InOutQueueControllerSystem

InOutQueueControllerSystem is a queue system controller component for queue systemts that contains both in and out queues. This class inherits the properties of SubSystem.

 

3.14.     Components in package com.teletalk.jserver.queue.messaging

3.14.1.   QueueMessagingManager

QueueSystemCollaborationManager implementation using messaging (com.teletalk.jserver.tcp.messaging.MessagingManager) as communication framework. This class inherits the properties of MessagingManager. Note that the property connectionsPerDestination is set to 1 and removed, since the implementation depends on the fact that only one connection exists to each destination.

 

3.15.     Components in package com.teletalk.jserver.rmi

3.15.1.   RmiManager

RmiManager is the core component enabling RMI communication with this server. This class inherits the properties of SubSystem.

Properties:
  • checkInterval - The interval in milliseconds at which this RmiManager should perform various checks to determine the status of, among other things, the connection to the Rmi registry. Default value is 5000 ms.
  • registryAddresses - Addresses (host:port) to RMI registries to which this RmiManager should connect and register the server. Default value is 0.0.0.0:1099. Note that on JVM versions prior to 1.4, only one address may be specified.

    • •         rmiCodeBase - The value for the system property java.rmi.server.codebase, used when loading resources like custom administration panels from this server via RMI. If multiple codebase urls are to be specified, they must be delimited by spaces. If a codebase url denotes a directory it must have a trailing /.

  • rmiMethodCallLoggingEnabled - Boolean value indicating if rmi method calls should be logged. If this property is set to true all method calls in the rmi adapter classes (package com.teletalk.jserver.rmi.adapter) will be logged. Default value is false.
  • rmiExportAddresses - The local addresses on which RMI objects (such as subsystem RMI adapter classes etc.) should be exported (listen) on. Note that only one address is possible at the moment. If no value is specified, the first available port on the local host will be used.

 

3.16.     Components in package com.teletalk.jserver.spring

3.16.1.   SpringApplicationContext

SpringApplicationContext is a component for Spring integration and inherits the properties of SubComponent.

SubComponent subclasses defined in the application context will be given the name of the bean (id or name) and registered under the top subsystem (JServer). If however the name of the bean is of a dot-separated format, SpringApplicationContext will attempt for find a matching name chain in the server and add the component under the appropriate parent. Components will be engaged automatically after creation.

All core systems in JServer will be made available as beans in a parent context of the context created by this component. The names of those beans correspond to the core component aliases defined in class JServerConstants, which are the same as the core component tag names (see chapter 2.5).

Properties:
  • applicationContextPath - The path of the Spring XML application context bean definition file to be used. Default value is spring.xml.

 

3.16.2.   SubComponentBeanProxy

SubComponent to be used as a proxy for a bean (typically contained in a Spring application context, although this is not necessary) and thus making it possible to expose the properties of that bean though JServer properties. The bean property types that may be controlled though this class are all primitive types and corresponding wrapper classes as well as the classes java.lang.String, java.io.File and java.net.URL.

The bean reference is set by calling the method setBean(Object).

Properties:
  • beanName - The name of the bean that is to be proxied.
  • useJServerPropertyFile - Flag indicating if the JServer property file should be used as soruce for property values.

 

3.17.     Components in package com.teletalk.jserver.statistics

3.17.1.   StatisticsManager

StatisticsManager is a component for runtime statistics of a server. This class inherits the properties of SubComponent.

 

3.18.     Components in package com.teletalk.jserver.tcp

3.18.1.   TcpServer

TcpServer is a simple tcp listener component that inherits the properties of SubSystem. TcpServer may use a TcpConnectionPool (usually named ConnectionPool) and a SSLServerSocketFactory as child components.

Properties:
  • localPort - The the port on which this TcpServer should listen for requests. The port number must be between 0 and 65535, inclusive.

    • •         localAddress - The local address to which this TcpServer is bound. If the value of this property is empty or '0.0.0.0' the server will accept connections on any/all local addresses.

  • backlog - The maximum queue length for incoming connection indications (a request to connect). The backlog must be >= 0. Default value is 50.
  • connectionLoggingEnabled - Flag indicating if connection activity should be logged. Default value is true.
  • serverSocketFactoryClass - Class name of the server socket factory to be created for this system on start up. An empty string (default value) means that the default factory will be used.

 

3.18.2.   TcpCommunicationManager

TcpCommunicatioManager is a component for tcp communication that inherits the properties of SubSystem. TcpCommunicatioManager may contain a number of TcpServers as child components (named “TcpServer@<ip:port>”, corresponding to the local addresses property). TcpCommunicatioManager also contains a ThreadPool (named EndpointPool) as a child component.

Properties:
  • activeEndpoints - A list containing information about all active endpoints.
  • endpointGroups - A list containing information about all endpoints groups.
  • localAddresses - The server addresses (ip:port) on which to listen for connections.
  • connectTimeout - Connect timeout in milliseconds used when creating new client sockets. Default value is 2500 ms.
  • connectAttempts - Maximum number of attempts that should be made to create a client socket connection before aborting. Default value is 5.
  • socketFactoryClass - Class name of the socket factory to be created for this system on start up. An empty string means that the default factory will be used.
  • serverSocketFactoryClass - Class name of the server socket factory to be created for this system on start up. An empty string means that the default factory will be used.
  • clientSocketBindAddress - The local address to be used when binding client sockets. Note that changing this property will not affect already created sockets. An empty value means that the default address will be used.
  • clientSocketBindPortBegin - The first port number to use when binding client sockets. Note that changing this property will not affect already created sockets. If this value is set to <= 0, client socket bind (local) port will be assigned automatically. Default value is -1.
  • clientSocketBindPortEnd - The last port number to use when binding client sockets. Note that changing this property will not affect already created sockets. If this value is set to <= 0, client socket bind (local) port will be assigned automatically. Default value is -1.

 

3.18.3.   SSLComponent

SSLComponent is an abstract base class for SSL enabled components. This class inherits the properties of SubComponent.

Properties:
  • securityProtocol - The name of the security protocol to use, for instance SSL or TLS. This property must be set.
  • securityProviderClass - The fully qualified name of an security provider class. Note that his property, if specified, takes precedence over the property securityProviderName. This property is optional. Default value is empty string, i.e. not set.
  • securityProviderName - The name of the security provider to use. If not specified, the default (or other matching provider) is used. This property is only checked if securityProviderClass is not set. Default value is empty string, i.e. not set.
  • keyManagerFactoryAlgorithmName - The name of the key management algorithm to be used by the key manager. If not set, the default algorithm will be used. Default value is empty string, i.e. not set.
  • keyStoreType - The type of key store to use, for instance jks or pkcs12. If not specified, the default type will be used. Default value is empty string, i.e. not set.
  • keyStoreName - The name (file name) of the key store to be used. This property must be specified if a key manager is to be used. Default value is empty string, i.e. not set.
  • keyStorePassword - The password set for the key store. Default value is empty string, i.e. not set.
  • keyManagerClass - The fully qualified name of a class to be used as a key manager. Default value is empty string, i.e. not set.
  • trustManagerFactoryAlgorithmName - The name of the trust management algorithm to be used by the trust manager. If not set, the default algorithm will be used. Default value is empty string, i.e. not set.
  • trustStoreType - The type of trust store to use, for instance jks or pkcs12. If not specified, the default type will be used. Default value is empty string, i.e. not set.
  • trustStoreName - The name (file name) of the trust store to be used. This property must be specified if a trust manager is to be used. Default value is empty string, i.e. not set.
  • trustStorePassword - The password set for the trust store. Default value is empty string, i.e. not set.
  • trustManagerClass - The fully qualified name of a class to be used as a trust manager. Default value is empty string, i.e. not set.
  • useNaiveTrustManager - Boolean value indicating if a naive trust manager is to be used, i.e. a trust manager that allows everything. If this property is set to true, all other trust store/manager related properties are disregarded. Default value is false.

3.22.     Components in package com.teletalk.jserver.tcp.messaging

3.22.1.   MessagingManager

MessagingManager is a component for high level messaging based communication between servers. This class inherits the properties of TcpCommunicationManager. MessagingManager contains a ThreadPool (named MessageHandlerPool) as a child component.

Properties:
  • remoteAddresses - Addresses (ip:port) for the destinations that the MessagingManager should attempt to create connections to. Note that address ids (interface names) also may be specified for each address through this property, by prefixing the address by the id and an equals sign, i.e. name=ip:port. This is also true for the property localAddresses, which is defined by the class TcpCommunicationManager.
  • connectionsPerDestination - The maximum number of client endpoints that are to be created for each destination. Default value is 2.
  • responseTimeout - Maximum time(ms) to wait for a response to a message sent to a remote messaging system. Default value is 10000 ms.
  • asynchMessageDispatchTimeout - Maximum time(ms) to wait for an asynchronous message to get dispatched. Default value is 10000 ms.
  • messageReadTimeout - The timeout in milliseconds used to detect stalled/aborted reading of incomming messages by a message receiver. Default value is 30000 ms.
  • checkInteval - The interval in milliseconds at which checks are performed on connections and meta data update commands are dispatched.. Default value is 30000 ms.
  • reportLoad - Boolean value indicating if a load value should be should be set automatically for each thread handling an incomming message (note that a load manager also must be present to enable load reporting). Default value is true.
  • enableServerAdministration - Boolean value indicating if remote server administration of this server should be enabled though this subsystem. Default value is true.
  • destinations - A list containing information about all destinations. This property replaces (renames) the property endpoint groups of TcpCommunicationManager.
  • statisticsEnabled - Boolean value indicating if statistics should be enabled. Default value is false.
  • proxyingEnabled - Boolean value indicating if proxying, i.e. if the MessagingManager should act as a proxy for all connected servers (or actually the registered named receivers of them), should be enabled. Default value is false.
  • remoteServiceNames - The names of the services that this MessagingManager is to connect to. Addresses for the service names will be fetched from an SNS.
  • localServiceNames - The names of the services provided by this MessagingManager, used for registratration in an SNS.
  • exposeMessageReceiversAsServices - Boolean value indicating if message receivers should be exposes as SNS services.
  • useProxiedMessageReceivers - Boolean value indicating if message receivers available via a proxy should be used for message dispatch. This property will affect the destinations returned by the methods getDestinations(String) and getDestinations(Map).

3.25.1.   DiskSpaceCheckAction

DiskSpaceCheckAction is a periodic action that performs disk space checks. This class inherits the properties of PeriodicAction. Note: This class relies on a native implementation which for the moment only is available for Windows.

Properties:
  • drives - The drives (paths) and corresponding minimum disk free space in MB (for instance C=1000). Default value is null, i.e. not set.
  • lowDiskSpaceErrorCode - The critical error code that will be logged when a drive/path has too low disk space free. Default value is 9000 (the constant JServerConstants.LOG_MESSAGE_ID_LOW_DISK_SPACE).
  • diskSpaceCheckErrorsCritical - Boolean property indicating if disk space check errors should be logged as critical/fatal. Default value is false.
